# VaultApp — Android Secure Storage

A minimal Android app demonstrating production-grade secure secret storage using Android Keystore, AES-256-GCM encryption, and biometric authentication. Built as a portfolio project targeting MASVS L1/L2 compliance.

---

## Threat Model

| Asset | Attacker | Vector | Countermeasure |
|---|---|---|---|
| Stored secrets | Malware on same device | Read SharedPreferences | EncryptedSharedPreferences + AES-256-GCM |
| Encryption key | Root access / key extraction | File system read | Android Keystore hardware-backed key |
| App at rest | Reverse engineer | APK decompilation | R8 obfuscation (release build) |
| Running app | Attacker with physical access | Frida / debugger | Root detection, emulator detection, integrity check |
| Secrets in memory | Memory dump | Runtime inspection | Keys never leave Keystore, decryption on demand only |

## Security Architecture

### Encryption

All secrets are encrypted with **AES-256-GCM** before storage. The encryption key lives exclusively in the **Android Keystore** and never leaves it — cryptographic operations happen inside the Keystore, the raw key bytes are never exposed to the application layer.

```
User secret
    │
    ▼
AES-256-GCM encryption (Cipher initialized via Keystore)
    │
    ├── IV (12 bytes, generated by Keystore)
    └── Ciphertext
    │
    ▼
Base64(IV + Ciphertext) stored in EncryptedSharedPreferences
```

The storage layer adds a second layer of encryption via `EncryptedSharedPreferences` (AES-256-GCM for values, AES-256-SIV for keys), backed by a separate Keystore key managed by Jetpack Security.

### Biometric Authentication

`BiometricPrompt` is used with a `CryptoObject` — the biometric authentication is cryptographically bound to the encryption/decryption operation. This means:

- The Keystore cipher cannot be initialized without successful biometric authentication
- Frida hooking `onAuthenticationSucceeded` does not bypass the protection — the TEE (Trusted Execution Environment) will not unlock the cipher without a real biometric match
- Each encrypt/decrypt operation requires fresh biometric authentication (`setUserAuthenticationParameters(0, AUTH_BIOMETRIC_STRONG)`)

### Key Properties

```kotlin
KeyGenParameterSpec.Builder(KEY_ALIAS, PURPOSE_ENCRYPT or PURPOSE_DECRYPT)
    .setBlockModes(BLOCK_MODE_GCM)
    .setEncryptionPaddings(ENCRYPTION_PADDING_NONE)
    .setKeySize(256)
    .setUserAuthenticationRequired(true)
    .setUserAuthenticationParameters(0, AUTH_BIOMETRIC_STRONG)
```

`setUserAuthenticationParameters(0, ...)` — the `0` means authentication is required on every single use of the key, not just once per time window.

### Resilience Controls

**Root Detection** — checks for `su` binary presence in known paths, known root package names (Magisk, SuperSU), build tags (`test-keys`), and writable `/system` partition.

**Emulator Detection** — checks build properties (`Build.FINGERPRINT`, `Build.HARDWARE`, `Build.PRODUCT`), known emulator hardware identifiers, and emulator-specific device files.

**Integrity Check** — verifies the app's signing certificate hash at runtime. In debug builds this check is bypassed intentionally; in release builds, a tampered or repackaged APK will be detected and the app will refuse to run.

## What I Intentionally Didn't Do (And Why)

**No NDK/JNI for crypto** — The Android Keystore already provides hardware-backed key storage. Moving crypto to native code would add complexity without improving the actual security boundary. The threat model here is key extraction, not algorithm reversing.

**No custom TrustManager** — Not applicable for a local storage app with no network calls. Certificate pinning will be covered in a separate project (Project 2 — Network Pinning Lab).

**No remote attestation** — Key attestation is verified locally. Full remote attestation (sending the attestation certificate chain to a server for verification) is out of scope for a local-only app but would be the next step in a production context.

**No anti-tampering beyond signature check** — More aggressive techniques (checksum of specific .dex files, native integrity checks) would add friction without materially changing the threat model for this scope. The signature check covers the primary repackaging threat.
